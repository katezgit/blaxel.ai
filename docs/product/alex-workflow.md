# Alex's workflow

Six phases Alex moves through when she ships and operates an agent on Blaxel. [`personas.md`](./personas.md) tells you who she is; this file tells you what she is doing at each step and what the Blaxel dashboard must give her to keep her on the platform. Earlier phases gate later ones — you cannot debug a stuck Agent if you can't trust the Sandbox state machine, and you cannot reason about cost or capacity if Policies and API Keys aren't legible.

> **Conventions used in this doc:**
> - Phases are ordered by the order Alex *first* encounters them when standing up a new agent product on Blaxel. After the runtime is live, Alex re-enters phases out of order — most often Phase 4 (drive + observe) and Phase 5 (debug) — see "Re-entry" at the end of this file.
> - Each phase has three fields: **Does** (concrete actions in product vocabulary), **Why** (the failure mode this phase prevents or the win it enables), **Surface implication** (what the dashboard must do to support this phase).
> - Exactly one phase is the **load-bearing UX of the entire product** for this persona — called out in its Surface implication. Cuts preserve that phase first.
> - Every claim about Blaxel cites its source inline: `[Source: <file> § "..."]` for repo files, `[Source: <URL>]` for blaxel.ai / docs.blaxel.ai.

## Phase 1 — Pick or fork the Image

- **Does**: picks a platform-provided Image, or forks her own — takes a base Sandbox, installs deps + configures the environment + sets up the agent harness, then snapshots it as a reusable Image her team can spawn future Sandboxes from. Authoring happens in the CLI / SDK (`bl new sandbox my-sandbox` → `bl deploy`); the dashboard's Images list is where she confirms the snapshot exists, inspects the SHA, and copies the ID into IaC. [Source: platform.md § "Platform primitives" — *"Image … the customizable, reusable snapshot a Sandbox is forked from"*] [Source: https://docs.blaxel.ai/Sandboxes/Overview — *"bl new sandbox my-sandbox … bl deploy"*]
- **Why**: every customer session in production will spawn from this Image. If the Image is wrong — missing a dependency, wrong CUDA version, stale agent harness — the failure shows up in every Sandbox the platform spins after that point, not just one. Getting the Image right once is the leverage point for the entire runtime. Mental model: *git-fork, not container-boot* [Source: platform.md § "Platform primitives" — Image row].
- **Surface implication**: the Images list shows every Image in the Workspace with its SHA, region availability, last-spawn timestamp, and the `bl` command that produced it. Image detail surfaces the underlying configuration so Alex can diff a working Image against a broken one. **No "create Image" wizard** — authoring is CLI / SDK; the dashboard is inspection [Source: personality.md § "Sacrificial choices" #5 — *"CLI / SDK is a peer surface, not a fallback"*]. Empty state leads with the `bl new sandbox` command, not a `+ New` button.

## Phase 2 — Provision the runtime (Sandbox + storage)

- **Does**: spawns a Sandbox from the Image (region-pinned), attaches a **Volume** for state that must outlive the Sandbox, or mounts an **Agent Drive** if multiple Sandboxes / Agents need to read/write the same filesystem concurrently. Uses `SandboxInstance.createIfNotExists({ name, image, memory, region })` from a TS / Python script, or `bl` from the terminal. [Source: https://docs.blaxel.ai/Sandboxes/Overview — *"SandboxInstance.createIfNotExists({ name: 'my-sandbox', image: 'blaxel/base-image:latest', memory: 4096, region: 'us-pdx-1' })"*] [Source: platform.md § "Canonical user loops" #1 — *"Spin up the runtime … spawn a Sandbox from that Image (with attached Volume / Agent Drive)"*]
- **Why**: this is where the runtime's state model is decided. Volume vs Agent Drive is not interchangeable — Volume is regional block storage attached to one Sandbox, Agent Drive is distributed and concurrently mountable across many [Source: platform.md § "Disambiguation rules" — *"Volume is regional block storage attached to a Sandbox. Agent Drive is a distributed, concurrently mountable filesystem"*]. Picking the wrong one breaks the agent loop silently — a per-session Volume on what should be a shared Agent Drive will not let a peer agent pick up the conversation. Region pinning matters: a Volume attaching to a region-pinned Agent must match its region [Source: platform.md § "Platform primitives" — Volume row].
- **Surface implication**: the **Sandboxes** group of the IA is the steady-state landing for Alex. The Sandboxes list shows, per Sandbox: name + canonical ID (`sbx-7f3a9c1e`), state (`active` / `standby` / `deploying` / `deleted`), region, Image SHA, attached Volumes + Agent Drives, last-used timestamp, and the `expiresIn` countdown when a TTL is set [Source: platform.md § "Sandbox lifecycle states"]. Standby is a steady state — the UI must distinguish "your Sandbox is on standby" from "your Sandbox is deleted." Blurring them = FAIL [Source: platform.md § "Disambiguation rules" — *"Sandbox standby vs deleted"*]. Volume + Agent Drive sub-lists sit inside the Sandboxes group, peer to Sandboxes, so the relationship is one nav click away.

## Phase 3 — Deploy the workload (Agent / Batch Job / MCP Server / Model API)

- **Does**: deploys the workload that runs *on top of* the Sandbox substrate. For Alex's product (Cubic / Shortwave / hirethomas.ai shape), that's typically an **Agent** — long-running, session-scoped, co-located with the Sandbox + Drive + Model API access [Source: platform.md § "Platform primitives" — Agent row]. For fan-out workloads (re-indexing a corpus, processing a queue of jobs), she composes a **Batch Job** that spawns thousands of Sandboxes from the Image and collects results [Source: platform.md § "Workload shapes" — *"Spawn thousands of Sandboxes in seconds — wide parallel fan-out"*]. If the agent needs to expose tool surfaces to other agents, she deploys an **MCP Server** (15-minute hard cap per invocation [Source: platform.md § "Workload shapes" — MCP Server row]). Inference goes through a **Model API**.
- **Why**: this is where workload *shape* gets committed to. The four L2 workloads have opposite scale shapes: Agent is vertical-and-long (1 Sandbox per session, long-running), Batch Job is horizontal-and-short (N Sandboxes, finite), MCP Server is serverless with a 15-minute cap, Model API is a long-running inference endpoint [Source: platform.md § "Workload shapes (L2)"]. Picking the wrong primitive for the workload shape is the silent-cost-blowup failure mode: running what should be a Batch Job as a fleet of Agents burns the runtime budget; running what should be an Agent as MCP Server invocations hits the 15-minute cap mid-conversation.
- **Surface implication**: the **Hosting** group of the IA surfaces all four L2 workloads as peers. Each workload's detail page makes its shape visible — Agent shows session count + per-session lifetime; Batch Job shows fan-out width + Sandbox count per Job; MCP Server shows invocation count + the 15-minute cap as a visible constraint; Model API shows endpoint URL + token throughput. The shape is not decoration — it's how Alex picks the right primitive when she's two hours into a new workload. The four primitives stay named and adjacent, per the Composed personality axis [Source: personality.md § "Personality adjectives" — *"every screen reads as named primitives from platform.md … visibly adjacent, not nested"*].

## Phase 4 — Drive the Sandbox from local + observe runtime

- **Does**: this is Alex's *daily-operating* phase — what she does between deploys and incidents. Opens a local terminal, runs `bl connect sandbox <name>` to open an interactive terminal session into the running Sandbox ("similar to SSH" [Source: platform.md § "Surfaces of the Blaxel platform"]). From a TS / Python script in her IDE, she drives `sandbox.process.exec()`, `sandbox.fs.write()`, `sandbox.previews.create()` against `SandboxInstance` while the dashboard shows live state for that same Sandbox [Source: personas.md § "Alex — In scope" — *"the Sandbox is treated as a remote target the local IDE drives, not as a thing one clicks through a wizard to configure"*]. The dashboard is the *inspection* peer: which Sandboxes are warm vs cold, which Agents are healthy vs degraded, per-Sandbox resource usage, per-Agent log stream. Standby is the wedge she relies on — Sandboxes idle for ~15s drop to standby with no memory charge, then resume in ~25ms on the next call [Source: https://docs.blaxel.ai/Sandboxes/Overview — *"If a sandbox does not have an active connection, it automatically transitions to a warm standby mode (also called scale-to-zero). This transition happens in approximately 15 seconds"*] [Source: https://blaxel.ai — *"25ms Resume from standby"*].
- **Why**: the runtime is *steady-state*, and Alex's job is to keep it that way. The CLI / SDK is where she *acts*; the dashboard is where she *sees what the runtime is doing right now* without translating between two mental models of the same Sandbox [Source: personality.md § "North star paragraph" — *"the runtime is perpetual (no cold start) and co-located (Sandbox + Agent + Model API on one plane) — the dashboard's job is to make those two facts visible as state, every paint"*]. If the surface lies about state — shows a Sandbox as `active` when it's actually been deleted, or hides the fact that the warm pool is half what it should be — Alex stops trusting the dashboard and writes her own observability. Cold-start latency in the dashboard's own paints is the same betrayal as cold-start in the runtime [Source: personality.md § "Personality adjectives" — *"Perpetual … first paint is live state, streams attach without spinners"*].
- **Surface implication**: **this is the load-bearing UX of the entire product for Alex.** First paint of the dashboard is current state — warm/cold Sandbox counts, Agent health, Batch Job outcomes, with each glance number drillable in one click to the filtered list [Source: personality.md § "Primary surface" — *"Glance … Deep … Switching altitudes is one click"*]. Sandbox detail surfaces three things on the same scroll: (a) the operational state — resource series, recent processes, last events; (b) the connection methods — REST endpoint, built-in MCP server endpoint at `<SANDBOX_BASE_URL>/mcp` [Source: platform.md § "Sandbox connection methods"], Preview URLs with the `bl_preview_token` mechanism [Source: platform.md § "Sandbox connection methods" — Preview URL row]; (c) the security band — Policy scope the Sandbox was deployed under, API Key prefix that created it, recent audit lines. **No Security tab** — the security band is a peer on the same page [Source: personality.md § "Sacrificial choices" #3 — *"Single-page ops + security band, never a 'Security view' tab"*]. Every state value streams; no value requires a refresh to update [Source: personality.md § "Interaction principles" #3 — *"Status streams, never polls"*]. Every aggregate number links to its underlying primitives [Source: personality.md § "Interaction principles" #7].

## Phase 5 — Debug an incident

- **Does**: production p99 spiked, the warm pool is half what it should be, two Sandboxes are pinned past TTL, or an Agent deployed an hour ago is throwing on a tool call [Source: personality.md § "North star paragraph"]. Alex opens the dashboard already 30 minutes deep — she has the symptom from her own monitoring, she needs the *root cause* from Blaxel. Reads the Sandbox boot trace, the Agent log stream, the Batch Job event timeline; correlates against the Policy that may have denied a retry and the API Key prefix that initiated the call. Drills from an aggregate ("3 Sandboxes errored in the last 5m") to the three specific primitives and their logs.
- **Why**: Alex is measured on agent reliability and per-session latency [Source: personas.md § "Alex — Why"]. One opaque incident costs trust in the platform; two costs the contract. Every layer of the runtime must be reachable from the surface that summarized it — no black-box health scores, no "your Sandboxes are fine" hero hiding a degraded one [Source: personality.md § "Personality adjectives" — *"Transparent — every layer of the runtime is reachable from the surface that summarizes it"*]. The failure mode named in the persona: Alex writes her own harness and tells her team Blaxel is not the answer when ops surfaces hide root cause [Source: personas.md § "Alex — Why" — *"Leaves for a competitor when ops surfaces hide root cause"*].
- **Surface implication**: from any primitive row, the forensic surface is one click — Sandbox row → boot trace + resource series + recent processes; Agent row → log stream + last tool calls; Batch Job row → event timeline + per-Sandbox status; Policy row → resolution path + denied calls; API Key row → call history [Source: personality.md § "Interaction principles" #2 — *"One click, primitive to trace. Two clicks = FAIL"*]. Failures outrank successes in pixel area, contrast, and inline context [Source: personality.md § "Interaction principles" #5 — *"Failure outranks success"*]. Every event has a stable URL Alex can paste into Slack or her incident channel [Source: personality.md § "Interaction principles" #8 — *"Every primitive and every event is URL-addressable"*]. Cmd-K command palette as the primary navigation; the mouse is the secondary [Source: personality.md § "Interaction principles" #10]. Empty states for a quiet incident view never read as "nothing here yet" — they read as last-event timestamp + last-seen state.

## Phase 6 — Govern + gate (Policies, API Keys)

- **Does**: defines a **Policy** that constrains *where / how / how-much* a workload runs — gating location (region), flavor (hardware tier), and token usage for Model APIs, MCP Servers, and Agent deployments [Source: platform.md § "Tenancy and access model" — *"Policies 'program how and where your workloads are deployed on Blaxel'"*]. Provisions **API Keys** for authenticating callers; assigns **Roles** for workspace access control (`admin` / `member`) [Source: platform.md § "Tenancy and access model" — Role row]. Rotates a leaked key, audits which Policy denied a deploy, narrows a key's scope after a security review.
- **Why**: agents call models, run untrusted code, browse the web, hand off to other agents. Without governance, a misconfigured Policy lets a workload spin up in the wrong region (compliance failure) or burn the token budget (cost failure); without scoped API Keys, one leaked credential leaks the workspace. Alex doesn't spend disproportionate time here (that's Sam) — but she needs the *control* on her path, because at 2am during an incident she's the one rotating the key. **Policy ≠ Role ≠ API Key** — three distinct concerns, conflating any two = FAIL [Source: platform.md § "Disambiguation rules" — *"Policy vs Role vs API key"*].
- **Surface implication**: the **Security** group of the IA — API Keys, Policies — is a peer of Sandboxes and Hosting at the top level. Never collapsed under Settings or a gear icon [Source: personality.md § "Sacrificial choices" #4 — *"Three IA groups stay flat and equal"*]. Policy detail shows the resolution path (which rule fired, on which deploy, why it denied or allowed) — not a permissions matrix. API Key detail shows call history (what this key invoked, against which primitive, when), the prefix (`bl_pk_3f8c…` truncated visibly, full value pasteable [Source: personality.md § "Anti-personality drift map" — *"Exact → precision theater"*]), and a one-click rotate. Audit lines read like log lines — cause, decision, primitive ID — not compliance brochure copy [Source: personality.md § "Voice & copy" — *"Security tone"*].

## Re-entry

Phase order describes Alex's *first* lap through Blaxel — Image → Sandbox → Workload → Drive → Debug → Govern. After the runtime is live, the re-entry pattern is different:

- **Daily steady-state.** Phase 4 (drive + observe) is where Alex lives between incidents. The dashboard's first paint serves this phase first.
- **Incident.** Phase 4 → Phase 5 — Alex notices an anomaly in the glance view and drills into the forensic surface. The drill must be one click [Source: personality.md § "Interaction principles" #2].
- **Capacity / cost decisions.** Phase 4 → Phase 6 — Alex sees the warm pool is too wide or a Batch Job is burning tokens, then opens Policies to gate it. The Policy edit must surface from the workload's detail page, not require navigating back to the Security group.
- **New deploy.** Phase 1 → Phase 2 → Phase 3 — Alex rolls out a new Image (or Agent / Batch Job / MCP Server) and watches it through Phase 4. The launch itself is CLI / SDK; the dashboard surfaces the new primitive landing in the relevant list within seconds.

*Designing flows that skip a phase Alex is in — e.g., a setup wizard that intercepts her Phase 4 dashboard open with a Phase 1 onboarding card — breaks the workflow Alex trusts. The "Alex wizard creep" anti-pattern in [`personas.md`](./personas.md) is exactly this failure.*

*Phase order governs the first lap; the dashboard must support re-entry into any phase from any other phase without a navigation reset.*
