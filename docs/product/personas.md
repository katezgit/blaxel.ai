# Personas

Two personas, one primary, one secondary. The product is designed around the primary; the secondary is checked as a sanity gate, not optimized for.

> **Derivation note (2026-06-18).** Initial draft derived from Blaxel's public marketing positioning (perpetual sandbox platform), the current dashboard IA (Sandboxes / Hosting / Security), and the customer logo wall — *"Trusted by AI builders from high-growth companies"* (Cubic, Mendral, Shortwave, Polsia, Vybe, Casco, Bloom, Cartage, hirethomas.ai, Sapiom, Webflow, Strapi, Ploy, Human Behavior, **Build0, Corvera, CodSpeed**). Two customer sub-shapes are visible: (a) **AI-native growth-stage startups** whose entire product is an agent (Cubic = AI code review, Shortwave = AI email, hirethomas.ai = AI hiring agent, Vybe = AI voice, etc.) and (b) **established product companies** (Webflow, Strapi) bolting agent capabilities onto mature platforms. Alex maps to (a); Sam maps to (b). A **possible third shape** is visible in **Corvera** (CPG agentic workforce) — operator-deployed agentic workforce in a non-software industry — which doesn't cleanly fit Alex (AI-native infra) or Sam (established product co bolting AI on). Flag for investigation; do not force-fit. Fresh signals added: **Build0** (sandbox cost / scale-to-zero economics narrative), **CodSpeed** (code performance, developer tooling). These slots are derived, not interview-validated — flag mismatches when operator interviews land.

## AI Platform / Agent Engineer (Alex, 28) — PRIMARY

- **What**: Platform / infra engineer (or senior product engineer wearing the infra hat) at an AI-native growth-stage startup whose entire product is an agent — code-review agent, voice agent, hiring agent, email triage agent, etc. (the Cubic / Shortwave / hirethomas.ai / Vybe shape). Alex is responsible for the runtime under the agent: every customer session needs a fresh sandbox, every agent step might execute untrusted code or shell out to a tool, and cold-start latency is a product-quality blocker. Lives in the **SDK / CLI** for routine ops (spawning sandboxes, deploying agents, rotating API keys, mounting volumes) and opens the **dashboard** for forensics, capacity inspection, and ops the CLI can't do better — debugging a stuck Job, inspecting which Sandboxes are warm, auditing a Policy, tracing why a Custom Domain isn't routing. Skips welcome tours and config wizards — already deep in the problem when the dashboard opens.
- **When**: Bursty during a build or incident, steady-state during normal ops. Sessions are triggered by (a) a production incident — sandbox leaked, agent stuck, latency spiked, capacity saturated; (b) a deploy — rolling out a new Agent / Image / Job; (c) a capacity / cost decision — looking at which Sandboxes are warm and what they're costing. Usually mid-debug when opening the dashboard, often with the CLI and a terminal already open. No tolerance for friction between "I see the problem" and "I can act on it."
- **Why**: Measured on agent reliability, per-session latency (especially cold-start), unit economics, and security posture (one sandbox escape kills the company). Trusts platforms that show their work — concrete numbers, raw logs, real APIs. Distrusts platforms that hide what the agents did or how the sandbox booted. Pays Blaxel for the *perpetual* part (no cold start) and the *co-located* part (agent + sandbox + model on the same plane = low latency). Leaves for a competitor when ops surfaces hide root cause, when the abstraction breaks ("this is happening but I can't tell why"), or when the security model is opaque.
- **In scope for Alex**: Everything under **Sandboxes** (Sandboxes, Volumes, Agent Drive, Images), **Hosting** (Agents, Jobs, MCP Servers, Model APIs, Network, Custom Domains), and **Security** (API Keys, Policies). The SDK (`@blaxel/core`, entry class `SandboxInstance`) + CLI (`bl`) are Alex's primary daily surface; the dashboard is the forensics + capacity-decision surface. **Canonical operating pattern:** `bl connect sandbox <name>` from a local terminal opens a remote-dev-env channel into the running Sandbox, then Alex drives `sandbox.process.exec()`, `sandbox.fs.write()`, `sandbox.previews.create()` etc. from a local TS / Python script in VS Code while the dashboard shows live state — the Sandbox is treated as *a remote target the local IDE drives*, not as a thing one clicks through a wizard to configure. **Out of scope**: marketing-style onboarding flows, hand-holding wizards for primitives Alex already knows how to compose in code. Alex's composition lives in the SDK and IaC, not in form fields — the dashboard is for *seeing state* and *acting on incidents*, not for first-time-setup.
- **Workflow**: see [`alex-workflow.md`](./alex-workflow.md) for the phased journey — explore → provision runtime → deploy agent → run + observe → debug → scale + secure — with design implications per phase. *(Pending re-derivation against this persona — current `alex-workflow.md` is template state.)*

## Lead AI Engineer at an Established Product Co. (Sam, 34) — SECONDARY

Worked example below is the **Webflow / Strapi shape** — established product company shipping agent capabilities atop a mature platform. The same shape applies to other adjacent customers Blaxel serves (e.g. a mature SaaS adding a code-execution agent, a CMS adding a content-authoring agent).

> **Non-validation marker (2026-06-18).** Webflow + Strapi are **logo-only** on the marketing wall — no public case study, no quote, no narrative on blaxel.ai or docs.blaxel.ai confirms the "established product co bolting AI on" shape for either customer. Sam stays as a **hypothesis** worth sanity-gating designs against; stop citing the logo wall as evidence Sam is *real*. Hardening signals to seek: operator interview with the Webflow / Strapi integration lead, or a published Blaxel case study for either logo. (See "In scope for Sam" below for the indirect product-roadmap signals — HIPAA + Slack support bolt-ons — that suggest Sam is real but underbuilt.)

- **What**: Lead engineer or staff engineer leading the AI initiative inside an established product company (Webflow / Strapi shape — well-known product, large existing user base, AI is *new* to the product, not the entire product). Sam is integrating agent capabilities into an existing platform — needs to add a sandboxed runtime for user-generated code, agent-driven flows, or AI-authored output without rebuilding the sandbox stack. Reports to a CTO / VP-Eng who reads security / compliance posture before signing any vendor. Comfortable with the SDK but more likely to use the dashboard for integration setup, eval runs, and audit than Alex is.
- **When**: Project-driven, not bursty. Sessions cluster around integration milestones (initial setup, eval before rollout, security review, post-launch incident response). Less likely to be opening the dashboard mid-incident at 2am than Alex — more likely to be in a planned session with a checklist.
- **Why**: Measured on reliable, auditable AI features landing in a mature product without regressing the platform's core trust. Defensibility looks like: clean security model, clear policy surface for the auditor, predictable costs, no opaque magic. Audience for Sam's output is internal — eng leadership, security team, sometimes legal / compliance. The three concrete things Sam needs: (1) a security model they can explain to their security team (API Keys + Policies surface that maps cleanly to roles / scopes), (2) predictable cost + capacity (visibility into warm Sandboxes, jobs queued, etc.), (3) audit trails (who did what, when, against which primitive). Builds in-house when the vendor's security surface is too thin to defend in review.
- **In scope for Sam**: **Sandboxes**, **Hosting**, and especially **Security** (Sam spends disproportionate time here — API Keys, Policies, audit). The dashboard's security + audit surfaces are *more* important to Sam than they are to Alex. **Out of scope**: Alex-only deep ops surfaces — Sam doesn't optimize warm pool size, doesn't drill into individual Sandbox boot traces. Assumption: Sam's team is product engineers + a security person, not a dedicated platform-infra team. If a surface only matters during a 2am incident, it belongs on Alex's path, not Sam's.
- **Product-roadmap signals that Sam is real but underbuilt**: Blaxel sells **HIPAA compliance at $250/month** and **Slack support at $1,600/month plus 10% of usage** as **bolt-on add-ons**, not as a dedicated enterprise tier (source: https://blaxel.ai/pricing). These are the strongest public signals Sam exists in Blaxel's roadmap — enterprise compliance and high-touch support are Sam-shape concerns, not Alex-shape — but the fact that they ship as à-la-carte add-ons rather than as first-class surfaces (a "Compliance" hub, an "Enterprise" plan with these baked in) is itself the data point: **Sam is real but underbuilt in the current product**. The dashboard's current Security surface (API Keys + Policies) is the floor, not the ceiling, of what Sam will need.

## Anti-patterns

Persona mismatches caught in review. Re-check designs against this list before shipping.

### Sam "deep platform ops" creep
- **Misread**: Picturing Sam as the engineer who tunes warm pools, debugs sandbox boot traces, and reads raw Job event streams — i.e. doing all of Alex's work too.
- **Reality**: Real teams at the Webflow / Strapi shape have product engineers + a small security function, not a dedicated platform-infra team. They lean on the vendor for the deep ops surface; they own the security + integration surface. Confusing the two leads to dashboards that try to be a control plane for both audiences at once and serve neither.
- **Design implication**: Deep ops surfaces (warm-pool tuning, capacity inspection, raw boot traces, low-level Network config) belong on Alex's path. Sam's primary surfaces are Security (API Keys, Policies) and high-level Hosting (which Agents / Jobs are deployed, with which scope). If a flow only makes sense when you're already tuning warm pools, it belongs to Alex, not Sam.

### Alex "wizard creep"
- **Misread**: Adding Sam-style guided onboarding, setup wizards, or compliance-check overlays to Alex's path because "it doesn't hurt to have them" — they're skippable.
- **Reality**: Alex is mid-incident or mid-debug when the dashboard opens. Skippable wizards are not free — they delay the first useful screen, push real state below the fold, and burn Alex's attention. Alex already knows the primitive vocabulary; explaining "what is a Sandbox" wastes the first paint.
- **Design implication**: Keep Alex's surfaces free of guided-tour UI. The dashboard opens to *state*, not to *welcome*. First-time-setup affordances ship on a Sam-specific entry point (or behind an explicit "show me how" link), not on the default surface.

### "AI Builder" as a flat persona
- **Misread**: Treating Alex and Sam as a single persona ("AI builders") because both ship AI products. Designing for the union flattens both.
- **Reality**: Alex's failure mode is *opacity at 2am*; Sam's failure mode is *security posture they can't defend in review*. These produce opposite design constraints: Alex wants raw signal, Sam wants clean abstractions over the same data. The same screen can serve both only if the affordance is layered — primary state for Alex's eye, expandable detail / audit context for Sam's.
- **Design implication**: When a screen tries to serve both, *layer* — Alex-grade density at the top, Sam-grade context (policy scope, audit trail, security implication) as a lower band or expandable detail. Don't average the densities.

### Persona ≠ logo
- **Misread**: Reading the customer wall (Webflow, Strapi, Cubic, …) as a single archetype "AI builder" and designing for the average.
- **Reality**: The wall has two distinct shapes (AI-native startup, established product co.) with different team compositions, decision-making, and tolerance for friction. The shapes map cleanly to Alex / Sam — preserve the distinction.
- **Design implication**: When a design call is justified with "all our customers are AI builders," send it back. Re-state which persona the design serves and why the other one is the sanity check.
